1. Architecture Overview

The system consists of 5 specialized agents orchestrated by a webhook handler that listens to GitHub events. Each agent has a single, well-defined responsibility and communicates through GitHub's API (comments, status checks, labels).

Design principles

  1. Single responsibility — each agent does exactly one thing, well.
  2. Event-driven — agents react to GitHub webhooks, not polling.
  3. Stateless — agents don't remember previous runs. All context comes from the PR.
  4. Stack-aware — agents detect which stack changed and apply the right rules.
  5. Non-blocking — agents run in parallel. Only the webhook handler orchestrates.
  6. Fail-safe — if an agent fails, the PR is not blocked (except for required checks).

2. The 5 Agents

Agent 1 — Architecture Guard

Required Check Trigger: PR opened/sync Autonomous

Purpose: Enforces layer separation rules across all stacks. This is the most critical agent — it prevents architectural drift.

What it checks

StackRuleImplementation
PythonAPI layer must not import from repositories or ORMAST analysis or grep on import statements
PythonDomain layer must not import from APIGrep for from api import in models/
JS/TSComponents must not make direct API callsGrep for fetch(, axios. in components/
JS/TSNo console.log in production codeGrep excluding test files
TLPPendpoint.tlpp must not contain DB operationsGrep for DbSelectArea, RecLock, MsExecAuto
TLPPvalidation.tlpp must not import mapping or queryGrep for cross-layer function calls
AllNo hardcoded secretsRegex for API keys, tokens, passwords

Output

Posts a GitHub check run with status success or failure. On failure, adds inline annotations pointing to the exact line that violates the rule.

Agent 2 — Code Reviewer (AI-Powered)

Advisory Trigger: PR opened/sync Autonomous

Purpose: Uses Claude API to perform an intelligent code review. This is the "senior developer" that never sleeps.

What it reviews

  • Code quality and readability
  • Potential bugs and edge cases
  • Clean code adherence (naming, function size, single responsibility)
  • Missing error handling
  • Performance concerns (N+1 queries, unnecessary loops)
  • Suggestions for improvement with code examples

How it works

  1. Fetches the PR diff via GitHub API
  2. Detects the stack from file extensions
  3. Loads stack-specific review guidelines (from this document)
  4. Sends to Claude API with a structured prompt
  5. Posts the review as PR comments (inline on specific lines)

Output

Posts PR review comments. Does NOT block the merge — it's advisory. Human reviewers make the final call.

Agent 3 — Test Runner

Required Check Trigger: PR opened/sync Autonomous

Purpose: Runs the test suite and reports coverage. Blocks merge if tests fail or coverage drops below threshold.

Stack-specific behavior

StackTest commandCoverage threshold
Pythonpytest --cov --cov-fail-under=7070% lines
JS/TSjest --coverage70% lines, 60% branches
TLPPValidate JSON payloads + optional REST testsAll endpoints covered
Node-REDnpm test (if custom nodes exist)Custom nodes covered

Output

GitHub check run with pass/fail status. Posts a coverage summary comment on the PR.

Agent 4 — Security Scanner

Required Check Trigger: PR opened/sync Autonomous

Purpose: Scans for security issues: leaked secrets, vulnerable dependencies, dangerous patterns.

What it scans

  • Secrets detection — API keys, tokens, passwords in code (via trufflehog or gitleaks)
  • Dependency audit — known CVEs in dependencies (pip audit, npm audit)
  • TLPP-specific — hardcoded credentials in cPassword, cToken variables
  • Dockerfile — running as root, using latest tags, missing health checks

Output

GitHub check run. Blocks merge if secrets are found. Advisory for dependency warnings.

Agent 5 — PR Summarizer (AI-Powered)

Advisory Trigger: PR opened

Purpose: Automatically generates a human-readable summary of what the PR does. Helps reviewers understand the change before diving into code.

What it generates

  • One-paragraph summary of the change
  • List of files changed with a brief description per file
  • Risk assessment (low / medium / high) based on which layers are touched
  • Suggested reviewers based on CODEOWNERS and expertise

Output

Posts a comment at the top of the PR with the summary. Updates if the PR is updated.

3. Webhook Handler (Orchestrator)

The webhook handler is a lightweight Python service that receives GitHub webhook events and dispatches work to the appropriate agents. It runs as a Docker container.

Event routing


  GitHub Event                      Agents Triggered
  ─────────────────────────────────────────────────────
  pull_request.opened          →    ALL agents (parallel)
  pull_request.synchronize     →    Architecture Guard
                                    Code Reviewer
                                    Test Runner
                                    Security Scanner
  pull_request_review.submitted →   (none — human reviews only)
  issue_comment.created         →   Code Reviewer (if @netzero-bot is mentioned)

Stack detection logic

# The handler checks changed files to determine which stacks are involved
def detect_stacks(changed_files: list[str]) -> set[str]:
    stacks = set()
    for f in changed_files:
        if f.endswith('.py'):
            stacks.add('python')
        elif f.endswith(('.ts', '.tsx', '.js', '.jsx')):
            stacks.add('javascript')
        elif f.endswith('.tlpp'):
            stacks.add('tlpp')
        elif 'flows/' in f and f.endswith('.json'):
            stacks.add('nodered')
        elif f == 'Dockerfile' or f.startswith('docker-compose'):
            stacks.add('docker')
    return stacks

Deployment options

OptionProsCons
GitHub Actions (self-hosted runner)No extra infra, native GitHub integrationLimited control, cold starts
Docker on your own serverFull control, always warm, can use Claude API directlyNeed to manage infra
AWS Lambda / Cloud FunctionServerless, auto-scaling, cheap at low volumeCold starts, 15min timeout
Recommended: GitHub Actions as the primary runner For most agents, GitHub Actions is sufficient and requires zero additional infrastructure. The AI-powered agents (Code Reviewer, PR Summarizer) need API access to Claude, which works fine from a GitHub Action using repository secrets.

4. End-to-End PR Flow


  Developer pushes code
       │
       ▼
  Opens Pull Request
       │
       ├──→  PR Validation (GitHub Action — title, body, checklist)
       │
       ├──→  Webhook fires ──→  Orchestrator dispatches:
       │         │
       │         ├──→  [1] Architecture Guard  ──→  ✅ or ❌ (required check)
       │         ├──→  [2] Code Reviewer (AI)  ──→  📝 Review comments (advisory)
       │         ├──→  [3] Test Runner         ──→  ✅ or ❌ (required check)
       │         ├──→  [4] Security Scanner    ──→  ✅ or ❌ (required check)
       │         └──→  [5] PR Summarizer (AI)  ──→  📋 Summary comment
       │
       ▼
  All checks pass?
       │
       ├── No  ──→  Developer fixes issues, pushes again (loop back)
       │
       └── Yes ──→  Human reviewer examines:
                        ─ AI review comments
                        ─ PR summary
                        ─ The actual diff
                        │
                        ├── Approves ──→  Merge to main ──→  Deploy pipeline
                        │
                        └── Requests changes ──→  Developer fixes (loop back)

What blocks merge vs. what advises

AgentBlocks merge?Rationale
Architecture GuardYesLayer violations are structural debt — catch them before they enter main
Code Reviewer (AI)NoAI suggestions are helpful but not authoritative — human decides
Test RunnerYesBroken tests must never reach main
Security ScannerYes (secrets only)Leaked secrets are irrecoverable — everything else is advisory
PR SummarizerNoInformational only

5. Infrastructure & Deployment

Option A — All in GitHub Actions (simplest)

Every agent runs as a GitHub Actions workflow. AI agents use ANTHROPIC_API_KEY stored in repository secrets.

Option B — Central webhook + GitHub Actions hybrid

A lightweight webhook handler (FastAPI) receives events and triggers specific GitHub Actions via the repository_dispatch API. This gives more control over routing and allows rate limiting.

# Architecture:
#
# GitHub Webhook  →  FastAPI (Docker)  →  Repository Dispatch  →  GitHub Actions
#
# The FastAPI service is the "brain" that decides which agents to run.
# GitHub Actions are the "muscles" that do the actual work.

6. Using with Cowork / Claude Code

Cowork mode (interactive)

For interactive development sessions, you can invoke agents manually through Cowork:

  • "Review this PR" — triggers the Code Reviewer logic on the current branch diff
  • "Check architecture" — runs the Architecture Guard rules locally before pushing
  • "Summarize my changes" — generates a PR description draft
  • "Run security scan" — checks for secrets and vulnerabilities locally

Claude Code subagents

When using Claude Code directly, you can dispatch parallel subagents for larger tasks:

# Example: pre-push validation using subagents
#
# Main session dispatches 3 parallel subagents:
#   ├── Agent 1: Run architecture checks on changed files
#   ├── Agent 2: Run tests and report coverage
#   └── Agent 3: Scan for secrets and vulnerabilities
#
# All three run in parallel. Main session waits for results.
# If any agent reports issues, the developer fixes before pushing.

Autonomous agents via GitHub webhooks

The agents in this architecture can be set up to run autonomously — no human intervention needed. The webhook handler receives events from GitHub and dispatches the right agents automatically. This is the primary intended use.

The three modes of operation
  1. Autonomous (GitHub webhook) — agents run automatically on every PR. Zero effort from developers.
  2. Interactive (Cowork) — developer asks Claude to run specific checks during a session.
  3. CLI (Claude Code) — developer runs checks from the terminal before pushing.
All three modes use the same agent logic and the same rules. The only difference is how they're triggered.
