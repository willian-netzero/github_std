1. Overview & Prerequisites

What we are building

A standardized development workflow where every repository — regardless of stack — shares the same branching strategy, commit convention, code review process, and automated quality gates. The goal: any developer can switch between projects and immediately know how things work.

Current state vs. target state

TodayTarget
No consistent branching strategyGitHub Flow on all repos
Direct pushes to mainPRs required, with review + CI checks
No standard project structureLayered architecture per stack, enforced by templates
No automated testing or lintingCI pipeline blocks merge if quality gates fail
Ad-hoc code review (or none)Mandatory review with checklist
New repos created from scratchTemplate repos with pre-configured CI, structure, and docs

Prerequisites

  • GitHub Organization (netzero-hq) with at least a Team plan for branch protection rules
  • Admin access to the organization for the person executing this guide
  • Buy-in from all tech leads (present the Best Practices document first)
  • A pilot repository for each stack to test the setup before rolling out
Estimated total effort For one person dedicated to this: approximately 3–4 weeks to set up everything. For a team splitting the work: 2 weeks. The phased approach means value is delivered incrementally — you don't need to finish everything before teams start benefiting.

TLPP layer rules
  • endpoint.tlpp — Only handles HTTP concerns: reads the request, calls service, writes the response. Zero business logic here.
  • service.tlpp — Orchestrates the flow: calls validation, builds the mapping, calls ExecAuto or query. This is where the use case lives.
  • validation.tlpp — All input checks (required fields, type coercion, business rules like "status must be active"). Returns structured errors.
  • mapping.tlpp — The only place that knows about Protheus table/field names (SC1, SC7, SF1, etc.). Translates between JSON keys and Protheus field aliases.
  • query.tlpp — Only exists when reads are complex (multi-table joins, aggregations). Simple reads stay in the service.

Universal clean code rules (all stacks)

  1. No business logic in the entry layer. Controllers (Python), route handlers (JS), endpoints (TLPP) — they receive, delegate, respond.
  2. No database access outside the data layer. Services never write raw SQL or call ORM directly.
  3. One file = one responsibility. If you need "and" to describe what a file does, split it.
  4. Shared code belongs in shared/ or utils/. Never copy-paste between features.
  5. Configuration lives in environment variables. Use .env locally, GitHub secrets in CI, Protheus INI in TLPP.
  6. All public functions and endpoints must have documentation. Docstrings (Python), JSDoc (JS/TS), comments (TLPP).
  7. Dependencies flow inward. Domain never imports infrastructure. Mapping never imports endpoint.

5. Phase 4 — CI Pipelines (GitHub Actions)

4
Automate quality gates on every PR
~3 days

Every pull request must pass automated checks before it can be merged. This is non-negotiable — it removes the human factor from quality enforcement.

Step 4.1 — Create the universal PR validator

This goes in every repo and validates PR title (Conventional Commits) and description.

# .github/workflows/pr-validation.yml
name: PR Validation

on:
  pull_request:
    types: [opened, edited, synchronize]

jobs:
  validate-title:
    name: Conventional Commit Title
    runs-on: ubuntu-latest
    steps:
      - name: Check PR title format
        uses: amannn/action-semantic-pull-request@v5
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          types: |
            feat
            fix
            refactor
            chore
            docs
            test
            style
            perf
          requireScope: false
          subjectPattern: ^[a-z].+$
          subjectPatternError: |
            PR title "{subject}" must start with lowercase.
            Example: "feat(auth): add JWT refresh rotation"

  validate-body:
    name: PR Description Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/github-script@v7
        with:
          script: |
            const body = context.payload.pull_request.body || '';
            if (body.trim().length < 30) {
              core.setFailed('PR description is too short. Please fill the template.');
            }
            const checklistItems = (body.match(/- \[[ x]\]/g) || []);
            const checkedItems = (body.match(/- \[x\]/gi) || []);
            if (checklistItems.length > 0 && checkedItems.length === 0) {
              core.setFailed('Please check at least one item in the checklist.');
            }

Step 4.2 — Python CI

# .github/workflows/python-ci.yml
name: Python CI

on:
  pull_request:
    paths: ['**.py', 'pyproject.toml', 'requirements*.txt']

jobs:
  quality:
    name: Lint + Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install ruff mypy
      - name: Ruff lint
        run: ruff check src/ --output-format=github
      - name: Ruff format check
        run: ruff format --check src/
      - name: mypy type check
        run: mypy src/ --ignore-missing-imports

  test:
    name: Tests
    runs-on: ubuntu-latest
    needs: quality
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt && pip install pytest pytest-cov
      - name: Run tests with coverage
        run: pytest tests/ --cov=src --cov-report=term-missing --cov-fail-under=70

  architecture:
    name: Layer Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Verify no DB imports in API layer
        run: |
          VIOLATIONS=""
          if grep -rn "import.*sqlalchemy\|from.*repositories.*import" src/api/ 2>/dev/null; then
            VIOLATIONS="API layer imports from repositories/DB"
          fi
          if grep -rn "from.*api.*import" src/models/ 2>/dev/null; then
            VIOLATIONS="$VIOLATIONS | Domain layer imports from API"
          fi
          if [ -n "$VIOLATIONS" ]; then
            echo "::error::Layer violations found: $VIOLATIONS"
            exit 1
          fi
          echo "✓ No layer violations."

        run: |
          if grep -rn "fetch(\|axios\.\|\.get(\|\.post(" src/components/ --include="*.tsx" --include="*.jsx" 2>/dev/null | grep -v ".test.\|.spec."; then
            echo "::error::Components must not make direct API calls. Use services/ or hooks/."
            exit 1
          fi

Step 4.4 — TLPP CI

TLPP doesn't have a standard linter or test runner in CI, but we can still enforce structural rules and validate payloads.

# .github/workflows/tlpp-ci.yml
name: TLPP CI

on:
  pull_request:
    paths: ['**.tlpp', 'payloads/**', 'docs/**']

jobs:
  structure-check:
    name: Project Structure Validation
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Validate domain folder structure
        run: |
          echo "Checking TLPP domain folders..."
          ERRORS=""

          for domain_dir in src/*/; do
            domain=$(basename "$domain_dir")

            # Skip shared folder
            if [ "$domain" = "shared" ]; then continue; fi

            # Each domain must have at least endpoint.tlpp and service.tlpp
            if [ ! -f "$domain_dir/endpoint.tlpp" ]; then
              ERRORS="$ERRORS\nMissing: $domain_dir/endpoint.tlpp"
            fi
            if [ ! -f "$domain_dir/service.tlpp" ]; then
              ERRORS="$ERRORS\nMissing: $domain_dir/service.tlpp"
            fi
          done

          if [ -n "$ERRORS" ]; then
            echo -e "::error::Structure violations found:$ERRORS"
            exit 1
          fi
          echo "✓ All domain folders have required files."

      - name: Check for hardcoded credentials
        run: |
          if grep -rni "cPassword\s*:=\s*['\"].\+['\"]\|cToken\s*:=\s*['\"].\+['\"]" src/ 2>/dev/null; then
            echo "::error::Hardcoded credentials detected in TLPP source."
            exit 1
          fi
          echo "✓ No hardcoded credentials."

      - name: Check endpoint files have no business logic
        run: |
          # Endpoint files should NOT contain DbSelectArea, RecLock, MsExecAuto
          if grep -rn "DbSelectArea\|RecLock\|MsExecAuto\|MATA1[0-9][0-9]" src/*/endpoint.tlpp 2>/dev/null; then
            echo "::error::endpoint.tlpp must not contain DB operations or ExecAuto calls. Move to service.tlpp."
            exit 1
          fi
          echo "✓ Endpoint files are clean."

  payload-validation:
    name: Validate JSON Payloads
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate all JSON files
        run: |
          ERRORS=0
          for f in $(find payloads/ -name '*.json' 2>/dev/null); do
            if ! python3 -m json.tool "$f" > /dev/null 2>&1; then
              echo "::error file=$f::Invalid JSON: $f"
              ERRORS=$((ERRORS+1))
            fi
          done
          if [ $ERRORS -gt 0 ]; then exit 1; fi
          echo "✓ All payload JSONs are valid."

  docs-check:
    name: Documentation Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check README exists and is not empty
        run: |
          if [ ! -s README.md ]; then
            echo "::error::README.md is missing or empty."
            exit 1
          fi
          echo "✓ README.md exists."

What to test (priority order)

  1. Service / business logic layer — This is where bugs have the most impact. Unit test every service function.
  2. Validation layer — Test all validation rules with valid and invalid inputs.
  3. API contracts — Integration test that endpoints return correct status codes and response shapes.
  4. Edge cases — Null inputs, empty arrays, boundary values, concurrent access.

TLPP testing strategy

Since TLPP lacks a native unit test framework, we use a two-pronged approach:

1. Payload-based contract tests (automated in CI)

Every domain folder in payloads/ contains JSON files that represent valid and invalid requests. The CI pipeline validates their structure.

payloads/
├── purchase-request/
│   ├── valid-create.json        # Must return 200
│   ├── valid-update.json        # Must return 200
│   ├── invalid-missing-fields.json  # Must return 400
│   └── invalid-bad-status.json  # Must return 422
└── purchase-order/
    └── valid-create.json

2. Integration tests against AppServer (manual or scheduled)

A Python or shell script calls the TLPP REST endpoints with the test payloads and validates the responses:

# tests/integration/run-tests.sh
#!/bin/bash
BASE_URL="${APPSERVER_URL:-http://localhost:8080/rest}"

echo "=== Purchase Request Tests ==="

# Test: valid create
STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  -X POST "$BASE_URL/purchase-request" \
  -H "Content-Type: application/json" \
  -d @payloads/purchase-request/valid-create.json)

if [ "$STATUS" -ne 200 ]; then
  echo "FAIL: valid-create expected 200, got $STATUS"
  exit 1
fi
echo "PASS: valid-create → 200"

# Test: invalid payload
STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
  -X POST "$BASE_URL/purchase-request" \
  -H "Content-Type: application/json" \
  -d @payloads/purchase-request/invalid-missing-fields.json)

if [ "$STATUS" -ne 400 ]; then
  echo "FAIL: invalid-missing-fields expected 400, got $STATUS"
  exit 1
fi
echo "PASS: invalid-missing-fields → 400"

echo "=== All tests passed ==="

Key conventions for TLPP

File naming

Every domain folder uses exactly these file names. No variations.

FileResponsibilityImports fromNever imports
endpoint.tlppHTTP in/outservice, shared/restmapping, query, DB functions
service.tlppUse case orchestrationvalidation, mapping, query, shared/*HTTP parsing functions
validation.tlppInput + business checksshared/utilsmapping, query, DB functions
mapping.tlppJSON ↔ Protheus fieldsshared/protheus/datesservice, endpoint
query.tlppComplex readsshared/protheus/environmentendpoint, service

Standard response format

All TLPP endpoints must return this JSON shape (defined in shared/rest/response.tlpp):

// Success (200)
{
  "success": true,
  "data": { ... },
  "message": "Purchase request created successfully"
}

// Validation error (400 / 422)
{
  "success": false,
  "errors": [
    { "field": "C1_PRODUTO", "message": "Product code is required" },
    { "field": "C1_QUANT",   "message": "Quantity must be greater than zero" }
  ],
  "message": "Validation failed"
}

// Server error (500)
{
  "success": false,
  "errors": [],
  "message": "Internal server error — check AppServer logs"
}

ExecAuto wrapper pattern

Never call MsExecAuto directly. Use the shared/protheus/execauto.tlpp wrapper that handles error trapping:

// shared/protheus/execauto.tlpp — simplified pattern

Function NzExecAuto(cFunction, nOption, aFields, cMessage)
  Local lSuccess := .T.

  BEGIN TRANSACTION
    PREPARE ENVIRONMENT EMPRESA '01' FILIAL '01'

    MsExecAuto({|x,y,z| &cFunction(x,y,z)}, aFields, nOption)

    If lMsErroAuto
      cMessage := MostraErro("", "")
      lSuccess := .F.
      DISARM TRANSACTION
    EndIf
  END TRANSACTION

Return { lSuccess, cMessage }

Appendix C — Existing Repo Migration Checklist

For each existing repository that needs to adopt the new standards:

Pre-migration

  • [ ] Identify the stack and choose the matching template as reference
  • [ ] Review current folder structure and map files to the standard layers
  • [ ] Identify files that violate layer separation (e.g., SQL in controllers)
  • [ ] Check for hardcoded credentials or config values

Migration steps

  1. [ ] Create a migration/adopt-standards branch
  2. [ ] Add .github/workflows/ — copy from template repo
  3. [ ] Add .github/PULL_REQUEST_TEMPLATE.md
  4. [ ] Add .github/ISSUE_TEMPLATE/ (bug-report.yml + feature-request.yml)
  5. [ ] Add .github/CODEOWNERS
  6. [ ] Add .github/labeler.yml
  7. [ ] Add .editorconfig
  8. [ ] Update .gitignore from template
  9. [ ] Reorganize src/ to match standard layer structure
  10. [ ] Move tests to tests/unit/ and tests/integration/
  11. [ ] Add payloads folder with test JSONs (TLPP only)
  12. [ ] Update README.md with standard sections
  13. [ ] Add docs/ folder with development-guide, architecture, security, roadmap, and backlog templates
  14. [ ] Open a PR, let CI run, fix any failures
  15. [ ] After merge, enable branch protection on main

Post-migration

  • [ ] Verify CI runs on next PR
  • [ ] Verify branch protection blocks direct pushes
  • [ ] Verify CODEOWNERS assigns the right reviewers
  • [ ] Update any CI/CD deployment pipelines if paths changed
  • [ ] Notify the team that the repo now follows the new standards
Do NOT try to refactor code and migrate structure in the same PR. The migration PR should only move files and add CI config. Refactoring layer violations (like extracting DB queries from controllers) should be done in follow-up PRs, one at a time.
