1. Branching Strategy

We follow GitHub Flow — a simple, trunk-based model that works for all stacks. There is one long-lived branch (main) and short-lived feature branches.

main  ─────●────────●────────●────────●──────── (always deployable)
              \              \       ↑       ↑
               \              \     │       │
  feat/login  ──●──●──●──PR──┘       │
                         fix/api-timeout  ──●──PR──┘

Branch naming

PrefixPurposeExample
feat/New featurefeat/user-auth
fix/Bug fixfix/api-timeout
refactor/Code improvement (no behavior change)refactor/extract-service
chore/Tooling, deps, configchore/update-deps
docs/Documentation onlydocs/api-readme
test/Adding or fixing teststest/auth-unit-tests
hotfix/Critical production fixhotfix/null-pointer-crash
Rule: short-lived branches only. A feature branch should live at most 3–5 days. If a feature is large, break it into smaller PRs behind a feature flag.

Protected branch rules for main

  • Require pull request before merging (no direct push)
  • Require at least 1 approval
  • Require status checks to pass (lint + tests)
  • Require branch to be up-to-date with main
  • Require linear history (squash or rebase merge)

2. Commit Conventions

We use Conventional Commits to keep history readable and enable automated changelogs.

<type>(<scope>): <short description>

[optional body]

[optional footer(s)]

Types

TypeWhen to use
featA new feature visible to the user
fixA bug fix
refactorCode change that neither fixes a bug nor adds a feature
testAdding or updating tests
docsDocumentation changes
choreBuild process, dependencies, CI config
styleFormatting, missing semicolons (no logic change)
perfPerformance improvement

Examples

feat(auth): add JWT refresh token rotation
fix(api): handle null response from carbon-credit endpoint
refactor(dashboard): extract chart component to shared module
chore(deps): bump react from 18.2 to 18.3
test(billing): add integration tests for invoice generation
Good commits Each commit represents one logical change. If you need the word "and" in your message, it should probably be two commits.
Avoid fix stuff, WIP, changes, update — these say nothing about what changed or why.

3. Project Structure & Layer Segregation

Regardless of stack, every project must have clear separation of concerns. The layer names may change, but the principle stays the same: each layer has one responsibility and depends only on the layers below it.

General layered architecture

┌─────────────────────────────────────────┐
│           Presentation / UI             │  ← React components, views, CLI, Node-RED dashboard
├─────────────────────────────────────────┤
│           Application / Service         │  ← Use cases, orchestration, business rules
├─────────────────────────────────────────┤
│           Domain / Model                │  ← Entities, value objects, domain logic
├─────────────────────────────────────────┤
│           Infrastructure                │  ← DB, HTTP clients, queues, file system, external APIs
└─────────────────────────────────────────┘

  Dependency direction: top → bottom (never upward)

Clean Code — Non-Negotiable Rules

  1. No business logic in controllers/routes. Controllers receive the request, call a service, return the response. That's it.
  2. No database queries outside the repository/DAO layer. Services never import the ORM or run raw SQL.
  3. One file, one responsibility. A file named user_service.py should only handle user-related operations.
  4. Shared code goes in utils/ or common/. Never duplicate logic across features.
  5. Configuration lives in environment variables, not hardcoded strings. Use .env files locally, secrets in CI.
  6. All public functions and API endpoints must be documented — docstrings for Python, JSDoc for JS/TS, comments for TLPP.
  7. Dependencies flow inward. The domain layer never imports from the infrastructure layer.
Common violation Putting a database query inside a React component or an API route handler. If you see SELECT or db.query() in a controller, that's a code review rejection.

4. Pull Requests & Code Review

PR template (auto-applied via GitHub)

## What does this PR do?
<!-- Describe the change in 1–3 sentences -->

## Type of change
- [ ] feat — new feature
- [ ] fix — bug fix
- [ ] refactor — code improvement
- [ ] chore — tooling / deps
- [ ] docs — documentation
- [ ] test — tests only

## How was this tested?
<!-- Describe the tests you ran -->

## Checklist
- [ ] Code follows the layer separation guidelines
- [ ] No business logic in controllers/routes
- [ ] No hardcoded secrets or credentials
- [ ] Tests added or updated
- [ ] Linter passes with no warnings
- [ ] Self-reviewed my own code
- [ ] No console.log / print() left behind

Code review guidelines

The reviewer is the last line of defense before code reaches main. Review for these, in this order:

  1. Correctness — Does it do what the PR description says?
  2. Architecture — Are the layers respected? Is the right pattern used?
  3. Security — No exposed secrets, no SQL injection, no unvalidated input?
  4. Tests — Is the new/changed behavior covered by tests?
  5. Readability — Can another developer understand this in 6 months?
  6. Performance — Any obvious N+1 queries, unnecessary loops, or memory leaks?

Merge rules

ConditionRequired?
At least 1 reviewer approvalYes
All CI checks passing (lint + tests)Yes
PR description filled outYes
No unresolved review commentsYes
Branch up-to-date with mainYes
Squash merge (single clean commit)Recommended
Automated enforcement All of these rules are enforced by GitHub branch protection + status checks. Developers cannot bypass them.

5. GitHub Actions — CI/CD Pipelines

Every repository must have at minimum a PR validation pipeline that runs on every pull request. Below are ready-to-use workflows for each stack.

5.1 — Universal PR validation (all repos)

This workflow enforces that the PR title follows Conventional Commits and that the PR template checklist is filled.

# .github/workflows/pr-validation.yml
name: PR Validation

on:
  pull_request:
    types: [opened, edited, synchronize]

jobs:
  validate-pr-title:
    runs-on: ubuntu-latest
    steps:
      - name: Check Conventional Commit title
        uses: amannn/action-semantic-pull-request@v5
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          types: |
            feat
            fix
            refactor
            chore
            docs
            test
            style
            perf
          requireScope: false
          subjectPattern: ^[a-z].+$
          subjectPatternError: |
            The PR title "{subject}" must start with a lowercase letter.

  validate-pr-body:
    runs-on: ubuntu-latest
    steps:
      - name: Check PR description is not empty
        uses: actions/github-script@v7
        with:
          script: |
            const body = context.payload.pull_request.body || '';
            if (body.trim().length < 20) {
              core.setFailed('PR description is too short. Please fill out the template.');
            }

5.2 — Python (lint + test)

# .github/workflows/python-ci.yml
name: Python CI

on:
  pull_request:
    paths:
      - '**.py'
      - 'pyproject.toml'
      - 'requirements*.txt'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          pip install ruff mypy

      - name: Lint with Ruff
        run: ruff check src/ --output-format=github

      - name: Type check with mypy
        run: mypy src/ --ignore-missing-imports

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov

      - name: Run tests
        run: pytest tests/ --cov=src --cov-report=term-missing --cov-fail-under=70

  architecture-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check layer violations
        run: |
          echo "Checking for database imports in API layer..."
          if grep -rn "import.*sqlalchemy\|from.*models.*import" src/api/ 2>/dev/null; then
            echo "::error::API layer must not import database models directly. Use services."
            exit 1
          fi
          echo "No layer violations found."

5.3 — JavaScript / TypeScript (lint + test)

# .github/workflows/js-ci.yml
name: JS/TS CI

on:
  pull_request:
    paths:
      - '**.js'
      - '**.ts'
      - '**.tsx'
      - '**.jsx'
      - 'package.json'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Lint
        run: npx eslint src/ --max-warnings=0

      - name: Type check
        run: npx tsc --noEmit

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Run tests
        run: npx jest --coverage --coverageThreshold='{"global":{"branches":60,"functions":70,"lines":70}}'

  architecture-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check for console.log in production code
        run: |
          if grep -rn "console\.log\|console\.debug" src/ --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" | grep -v "// allowed" | grep -v ".test." | grep -v ".spec."; then
            echo "::error::Remove console.log/debug from production code."
            exit 1
          fi
          echo "No console.log violations found."

5.4 — Docker build validation (all repos with Dockerfile)

# .github/workflows/docker-validate.yml
name: Docker Validation

on:
  pull_request:
    paths:
      - 'Dockerfile'
      - 'docker-compose*.yml'
      - '.dockerignore'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Lint Dockerfile
        uses: hadolint/hadolint-action@v3.1.0
        with:
          dockerfile: Dockerfile

      - name: Build image (no push)
        run: docker build -t test-build:pr-${{ github.event.number }} .

5.5 — Auto-label PRs by path

# .github/workflows/auto-label.yml
name: Auto Label

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}

7. Quick-Reference Checklist

Before starting work

  • Pull the latest main
  • Create a branch with the correct prefix (feat/, fix/, etc.)
  • Check if there's an issue linked to your task

While coding

  • Keep business logic out of controllers/routes
  • Keep database access in the repository/DAO layer
  • Write tests alongside your code, not after
  • Commit small and often with Conventional Commit messages
  • No secrets, no hardcoded URLs, no console.log

Before opening a PR

  • Run the linter locally
  • Run tests locally and confirm they pass
  • Rebase on top of main if your branch is behind
  • Fill out the PR template completely
  • Self-review your diff one last time

As a reviewer

  • Check architecture and layer separation first
  • Verify tests actually cover the new behavior
  • Look for security issues (exposed secrets, unvalidated input)
  • Be constructive — suggest solutions, not just problems
  • Approve only when CI is green and you're confident
