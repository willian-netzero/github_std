Documentation strategy for NetZero

As NetZero scales industrially and geographically, documentation is becoming a strategic asset, not an administrative afterthought. This document proposes a scalable, future-proof documentation strategy that reduces operational and compliance risk, lowers long-term cost and friction, and positions NetZero to efficiently leverage AI in its operations.

Premises

Methodology

This document is based on the following premises:

  1. Doing something just because everyone else does it is not acceptable. Reasoning must be based on first principles.
  2. Looking for inspiration outside of NetZero's industry is paramount.
  3. Actively fighting psychological biases (in particular sunk costs, costs of transition, and siloed thinking) is key to creating and maintaining a competitive-advantage position.
  4. Preparing for the profound, structural transformations arising from AI developments is critical for companies to win. The power of embedding AI models (LLMs, vision, etc.) in business operations, optionally paired with robotics, must not be underestimated.
  5. Identifying (and fixing) early the business bottlenecks to scaling is a must-have discipline and a major differentiator.

Definitions

This work considers documentation in a narrow way. Documentation is understood as any document written with the aim to formalize rules or explain how rules should be followed. Documentation necessarily implies standardization, a need for constant up-to-date content, and availability of this content to all relevant people in the organisation.

Documentation is to be differenciated from (technical) specifications:

  • Specifications describe characteristics of an item (e.g., schematic of a reactor), whereas documentation explains how to use this item (e.g., operating procedures for the reactor).
  • Documentation can include specifications in its content, but there is always more to it.

Documentation is also to be differentiated from training material. Training materials are used to make sure that the relevant people have been informed about (and have understood) the documentation. Training courses are managed in a dedicated platform called NetZero Academy.

Last, documentation is included in the broader category of documents. The approach explored below does not consider documents other than documentation.

Documentation is typically composed of three types of objects, which are frequently combined:

  1. Text, which can be:
    • Plain text (paragraphs or formatted text: headings, bold, italics, etc.)
    • Tabular text (tables, bullet points, etc.)
  2. Diagrams, which can be:
    • Authoritative (they are the rule)
    • Illustrative (they exemplify the rule)
  3. Images, which can be:
    • Photographs (e.g., to illustrate operating procedures)
    • Technical drawings (e.g., from specification files)

Hereafter, it is assumed that:

  • Documentation is already digital and will remain so
  • Documentation is human-oriented; it is not meant for deterministic robots to execute upon, but could be leveraged by autonomous robots (e.g., Optimus) in the future
  • Documents other than documentation (e.g., commercial proposals, contracts, technical drawings, etc.) would be managed separately, for example: editable versions in their respective platforms (Microsoft, 3DS, AutoCAD, etc.) and final versions (PDFs) in the CRM, the ERP, or specialised storage. This should be further investigated and defined as part of a broader document management strategy for NetZero.
  • Documentation tied exclusively to a platform (e.g., Foundry) would be managed in said platform, although it may be worth exploring mid-term ways to export/convert it as well.

Current situation

NetZero currently has no globally defined strategy nor generalised practices in documentation management. Many things are yet to be documented and, when documentation exists, not all teams do and managed it the same way.

The ESG team has de facto taken leadership on a significant share of documentation efforts. It administers a SharePoint library which holds documentation from the following teams: OHS, Environment, O&M, HR, and Finance. This acts as a central repository, but:

  • files, structure, and access rights are entirely managed by hand
  • documentation from R&D, Engineering, and IT are not in scope
  • overall the current approach is not scalable

This situation leads to:

  • Messy storage

    "I don't remember where this document is saved."
    "I am not sure this is the main version or a copy."

  • Poor version control

    "I am not sure if this is the latest version of the procedure."
    "I couldn't find who approved this version."
    "When did we introduce the changes to comply with the new Puro methodology?"

  • Time-consuming manual update work

    "I need to reorganise all my diagram boxes to add a new step in my procedure."
    "I need to change one by one the text of each box of this diagram to translate it to Portuguese."

  • Costly software licenses (or similar)

    "Each user needs to pay 16 EUR / month for using Visio; I could use this money for other things."

  • Poor permissions management

    "I forgot to give access to the new site manager."
    "I realized an intern had undue access to R&D procedures."

The current situation must therefore be improved quickly and structurally to avoid major hurdles in the future, especially related to franchise and geographic expansion.

Target situation

In the target situation, any NetZero employee, franchisee, or contractor can, within seconds, access the authoritative version of the rules that apply to them, understand when they were last updated, who approved them, and ask a natural-language question to get a precise, source-linked answer.

Additionally, for public documentation, any edit should be automatically synced to the website so as to avoid versioning issues.

Therefore, the future documentation system must, by design:

  • guarantee a clearly identifiable single source of truth
  • provide intrinsic version control and approval traceability
  • enforce structure rather than rely on discipline
  • allow fast, low-cost updates, so that documentation does not lag operations
  • remain machine-readable and AI-accessible
  • avoid dependency on a single proprietary vendor
  • support text, diagrams, and images consistently

Solution

In industrial companies, the traditional approach to documentation is to implement a document management system (DMS).

However, these systems can be considered legacy and insufficiently scalable for NetZero's needs given that they:

  • Are very expensive
  • Are complex to implement and maintain
  • Require user discipline, as it adds a step in the process ("after the document is created, I need to remember to copy it to the DMS")
  • Are difficult to search given the variety of file types stored in them
  • Cannot enforce the consistent use of templates at the time of creating documentation
  • Usually result in massive replication of files across systems, leading to storage inefficiency and user confusion

The right solution for NetZero is to manage all documentation as code – an approach coined "docs-as-code", inspired by the software industry. This solution meets all the requirements described in Target situation.

The end-to-end solution can be split between backend (editing/management) and frontend (viewing only).

Backend

General

The backend platform to manage content shall be GitHub, retained over other solutions for its universality.

Remember: we will onboard new staff, external consultants, etc., so the tool needs to be familiar to the widest possible audience.

We would start by using GitHub's Organisation Free plan, and potentially upgrade later as needed. Benefits of upgrading include SSO with auto-provisioning of users and forced ownership of repositories by company instead of users.

For non-IT staff working on writing documentation, GitHub could be paired with a middleware (TBD) or used in a simplified version in the GitHub desktop app. In any case, documentation governance should ensure that, for a given topic and audience, there must be exactly one authoritative document at any point in time.

Context for non-IT people
Although Git-based platforms are often associated with software development, they are in fact general-purpose systems for managing structured content throughout its entire lifecycle.

In practice, a Git-based system allows NetZero to manage, within a single environment:

  • Content creation – authors create or update documentation using predefined templates, ensuring a consistent structure from the start.
  • Content review – proposed changes are submitted for review, allowing designated reviewers to comment, suggest changes, and discuss modifications directly on the content.
  • Content approval – approvals are formalised through controlled validation steps, providing a clear record of who approved what and when.
  • Version control – every change is automatically tracked, allowing NetZero to see the full history of a document, compare versions, and revert changes if needed.
  • Templates and standards enforcement – templates and automated checks ensure that documentation follows NetZero’s rules by design, rather than relying on manual discipline.

As a result, creation, review, approval, versioning, and publication of documentation all happen in the same system, with full traceability and without relying on email loops, manual tracking, or parallel tools.

For text

Text is straightforward in a docs-as-code approach, since the source format is text-based.

The most efficient syntax for NetZero's needs is Markdown, one of the leanest, most universal, and most efficient markup languages in existence. Anyone can learn it in minutes and it allows for standardization of text formatting.

Below is an example of Markdown used for text formatting.

  • Source code:
    # Title

## Section

### Sub-section

#### Sub-sub-section

Standard text. *Italics* text. **Bold** text. ***Bold italics*** text.
  • Result:

    Title

    Section

    Sub-section

    Sub-sub-section

    Standard text. *Italics* text. **Bold** text. ***Bold italics*** text.

Markdown rendering can be customised with specific styles, suiting branding needs: font type, size, and colour, paragraph spacing, etc.

Markdown will be the base language used for documentation, within which diagrams and pictures (and, more generally, assets) can be inserted.

For diagrams

At first, diagrams may seem to be difficult to handle in a docs-as-code approach:

  • How to build them in code?
  • How to embed them in otherwise text-based content?
  • How to know when they are authoritative or illustrative?

Luckily, specialised languages exist for diagrams, and they support being embedded within Markdown files.

The two best languages meeting NetZero's needs are:

  • Mermaid, for simple diagrams
  • BPMN (Business Process Model and Notation; created as xml files), for more complex diagrams

NetZero's rule should be: Mermaid for illustrative diagrams, BPMN for authoritative diagrams. In case of inconsistency, authoritative BPMN version prevails.

For reference, the above languages can produce:

  • Flowcharts
  • Activity diagrams
  • Sequence diagrams
  • Pie charts
  • Sankey diagrams
  • Timelines
  • Gantt charts
  • XY charts
  • ...

Regarding BPMN:

  • Benefits include:
    • BPMN is an internationally recognised standard for diagrams and logic
    • Diagrams can be drawn in a visual using a free desktop software such as Camunda Modeler, which works offline and gives access to the underlying BPMN code
    • Software enforces BPMN conventions at interface level
    • Software checks if logical flows have errors
  • Limitations (at least with Camunda Modeler) include:
    • No support for templates
    • No possibility of formatting (size of boxes, font size, etc.)

Repository structure

To properly enforce governance rules (permissions, templates, locations, etc.), having a structured backend organisation is key. In GitHub (and more generally in software), the base organisation container is called a repository (or "repo"), which itself contains folders, subfolders, and eventually files (markdown, images, etc.).

Below is what the repository structure could look like in GitHub (including repos beyond the direct scope of user-oriented documentation, for reference). The structure is intentionally kept consistent across repos, to better enforce automated governance rules across all of NetZero's GitHub.

Header structure (hidden metadata)

Good repository structure is not enough to manage the complexity of documentation over time. NetZero will increasingly face examples such as:

  • certain procedures apply only to certain sites (or certain technology versions) and/or certain geographies
  • certain policies or standards can keep the global rules but require minor adjustments to local conditions to be fully compliant and/or effective

Such cases cannot be handled by folders without creating copies of files. To maintain only one authoritative copy, we can instead leverage metadata.

A hidden header can be included in each Markdown file where tag-like, cumulative-attribute information is stored using YAML, a data serialisation standard. Part of YAML can be populated automatically based on who approved the file when; the rest can be added through standardised forms (for IT: using issue forms in GitHub).

For IT: YAML headers are to be validated automatically in CI (schema validation), and pull requests cannot be merged if required fields are missing or invalid.

Frontend

Dedicated docs webiste

For base users to access documentation, a frontend is needed as they won't interact with GitHub.

This frontend acts as the "consumption" layer of NetZero's documentation, and should allow for:

  • standard viewing of the content (directory tree, rendering interface, etc.)
  • search through both standard search bar and a "chat" functionality (which requires adapted indexing and retrieval of content)
  • for authorised users, commenting on the content

Candidate systems include Docusaurus, Wiki.js, BookStack, GitBook, and Document360. Docusaurus (an open-source, free platform) is probably a good fit, but further research and benchmarking is needed. In some cases, an add-on may be needed to support AI chat (e.g., DocSearch "Ask AI" in Docusaurus) and comments (e.g., Hypothesis).

In any case:

  • Authoritative documentation (code) should not be modifiable in the frontend
  • The AI chat feature must respect document permissions (i.e., never leak restricted content across roles/entities) and must not use the content to train the underlying model
  • Platform should support a hybrid public and private display of documentation depending on files (e.g., most policies are public, deployment standards are private, some procedures are only for NetZero and not for franchisees, etc.)
  • Platform should support zoom-in/zoom-out on diagrams, and ideally direct code rendering

Note: that having strict separation between backend and frontend allows to change the frontend later if needed (backend is unlikely to change given universality of git systems).

Practical aspects

Benefits

Having a docs-as-code foundation makes it possible to run a wide range of extremely powerful use cases on top, notably by leveraging LLMs. Below are a few examples.

Benefits for people in charge of documentation:

  • Access full history of revisions and approvals

    Example 1: When did we last update this policy?
    Example 2: Who approved this change?
    Example 3: What changed between these two versions?

  • Automate multi-language documentation management

    Example 1: I'm opening a new subsidiary in Mexico: let's instantly translate all of NetZero's documentation to Spanish.
    Example 2: We just updated the Travel and Transportation policy: let's update all its translated versions at once.

  • Automate internal coherence and consistency management

    Example 1: I'm introducing a new procurement rule: is it in contradiction with any pre-existing applicable rules?
    Example 2: I have updated my Land Acquisition policy: which other standards should I update to remain compliant?
    Example 3: I have renamed a position in the organisation: let's instantly update all documents accordingly.
    Example 4: A draft of diagram was prepared by my analyst: let's quickly run it through an LLM to check for any mistakes or missing pieces.

  • Centrally assign policies to specific sites / geographies / users

    Example: We just added a co-generation module to one of our sites: let's make available the relevant O&M procedures to the local team.

  • Convert in any format

    Example: I need to publish the policy on our website: done instantly by asking an LLM to change from Markdown to HTML.

  • Create trainings automatically

    Example 1: I need to train site managers to this new procedure: create quizzes to make sure they have a sufficient understanding of the procedure.

Benefits for all users, in particular non-technical:

  • Chat with the documentation to quickly find an answer to their question

    Example 1: What's the process to reserve a car?
    Example 2: Who should I contact to solve this problem?

  • Access the documentation in their preferred language

    Example: I'm a US auditor reviewing NetZero Brazil's procedures – I can instantly select my language

Challenges

Implementing docs-as-code within an industrial organisation that has no prior familiarity with this approach comes with multiple challenges, including:

  • Training non-IT people to use GitHub (even with a middleware)
  • Migrating legacy documentation
  • Maintaining update discipline over time
  • Managing bottom-up initiatives (documentation proposed by non-technical staff)

What this project is not

Just to be clear:

  • this is not a wiki free-for-all-to-edit project
  • this is not documentation by AI alone
  • this is not an IT-only initiative
  • this is not a project to "look good on paper"

Adoption plan

Pilot phase

Select 3-5 people within the IT, ESG, and HR teams to try it in practice.

Try:

  • Training people
  • Having them create new documentation from scratch
  • Having them converting old documentation to code
  • Quizzing LLMs on new documentation corpus to assess reliability of answers
  • Defining what the governance of the target system could look like (users to train, approval rules and flows, technical solutions to enforce them, authoritative versions vs. translated versions, etc.)
  • Understanding how this proposal meets international standards like ISO requirements
  • Defining the stack to be provided to documentation builders (e.g.: VS Code with approved extensions, GitHub Desktop, etc.)

Global roll-out

To be considered after the pilot phase. If pilot phase is conclusive, all learnings from this phase should be included in the roll-out plan.

Roll out shall not be done all at once, but gradually, in cycles, and splitting responsibilities between converting legacy documentation and creating new documentation.
